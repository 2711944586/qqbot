# knowledge/inbox 使用说明

这里放管理员整理过的本地素材，先用 `/kb inbox` 做只读体检，再用 `/kb ingest` 生成候选，最后用 `/kb show` 和 `/kb commit` 写入主知识库。

推荐每个素材一个 `.md` 或 `.txt` 文件，写“摘要、场景、可用话术、来源链接”，不要塞整段直播转写、完整切片字幕或长篇原话。

建议格式：

```text
标题：玩机器礼物感谢语气样本
来源：https://...
日期：2026-06-07
类型：公开切片摘要 / 本地授权笔记 / 礼物拟态模板

场景：
- 观众送礼物后，先短促感谢，再顺手接一个 CS 梗。

摘要：
- 感谢不宜太正式，像直播间随口接话。
- 可以说老板大气、别刷了顶不住、这波够下一把起枪，但不要声称是原话。

可用话术：
- 感谢老板，这波经济直接拉满。
- 别送了别送了，我这边压力已经上来了。

禁用：
- 不写“这是玩机器原话”，除非你有可核验短引用。
- 不复制超过几句的原始字幕。
```

操作：

```text
/kb inbox
/kb ingest
/kb list
/kb show <候选ID>
/kb commit <候选ID>
```

`/kb inbox` 会跳过这个 README，只检查真正的 `.md` / `.txt` 素材文件；它会提示长转写、完整字幕、未核验原话、时效事实缺来源和缺少场景化结构等风险。

`/kb ingest full` 只用于你已经整理好的长摘要。公开视频只做索引和摘要，想增强风格就写“什么场景怎么反应”，不要把版权内容整段搬进来。
